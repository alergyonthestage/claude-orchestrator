# Vault Profile Real Isolation — Analysis

**Date**: 2026-03-24
**Status**: Analysis — pending design approval

**Note**: This is a supporting analysis document. The definitive design is in
`profile-isolation-design.md`. Some decisions in this analysis were revised
during the design discussion — the design document takes precedence.

**Scope**: Fix vault profile system to provide real file isolation
**Supersedes**: Tracking-only isolation model from Sprint 7-Vault `design.md` §5.3

> This document analyzes all aspects of migrating vault profiles from
> tracking-only isolation to real git-level isolation, where `profile switch`
> changes the files visible on disk.

---

## 1. Problem Statement

### 1.1 Current Behavior (Tracking-Only)

The Sprint 7-Vault implementation uses **tracking-only** isolation:

- `add` marks a resource in `.vault-profile` but does NOT `git rm` from main
- `remove` unmarks from `.vault-profile` but does NOT delete files
- `move` copies files to target branch + updates `.vault-profile` but does NOT
  `git rm` from source branch
- `switch` does `git checkout <branch>` but since files exist on all branches,
  nothing visually changes

**Result**: Users see all projects on all profiles. The isolation promise is
broken. The only place isolation is enforced is during `vault push/pull`
(selective staging), which means:
- Locally, switching profiles has no visible effect
- The user must trust that push/pull will handle isolation correctly
- There is no way to have a "clean workspace" per profile

### 1.2 Additional Bugs

| # | Bug | Cause |
|---|-----|-------|
| B1 | `remove` crashes on second invocation | `git commit` without `|| true` on clean tree (line 1540) |
| B2 | `remove` message "now shared" is misleading | Resource was never removed from main |
| B3 | `move` doesn't remove from source | Missing `git rm` after copy |

### 1.3 Expected Behavior

After fixing, `vault profile switch` should change the files on disk:

```
# On profile "org-a":     projects/org-a-api/ exists, projects/personal-blog/ does not
# On profile "personal":  projects/personal-blog/ exists, projects/org-a-api/ does not
# On main:               only shared resources (global/, templates/, shared packs)
```

This leverages git's native branch checkout behavior — files that exist only
on one branch appear/disappear when switching branches.

## 2. File Categories in a Project

Understanding which files are git-tracked vs. untracked is critical for
designing correct move/copy/remove operations.

### 2.1 Complete File Map

```
projects/<name>/
├── .claude/                          # TRACKED — vault-committed
│   ├── CLAUDE.md
│   ├── rules/*.md
│   ├── agents/*.md
│   ├── skills/*/
│   ├── .cco/
│   │   ├── base/                     # TRACKED — merge ancestors
│   │   └── pack-manifest             # GITIGNORED
│   └── docs/                         # TRACKED
├── project.yml                       # TRACKED — project config
├── setup.sh                          # TRACKED — optional
├── mcp-packages.txt                  # TRACKED — optional
├── memory/                           # TRACKED — vault-committed (auto memory)
│   ├── MEMORY.md
│   └── <topic>.md
├── .cco/                             # MIXED
│   ├── source                        # TRACKED — origin tracking
│   ├── claude-state/                 # GITIGNORED — session transcripts
│   ├── docker-compose.yml            # GITIGNORED — generated
│   ├── managed/                      # GITIGNORED — runtime
│   └── meta                          # GITIGNORED — schema version
├── .tmp/                             # GITIGNORED — dry-run artifacts
├── rag-data/                         # GITIGNORED — RAG data
└── secrets.env                       # GITIGNORED — secrets
```

### 2.2 Categories for Profile Operations

| Category | Examples | On `git checkout` | Profile ops must handle? |
|----------|----------|-------------------|--------------------------|
| **Tracked** | `.claude/`, `project.yml`, `memory/` | Appear/disappear automatically | No — git handles it |
| **Gitignored, portable** | `.cco/claude-state/`, `memory/` (pre-migration) | Stay on disk (git ignores) | **Yes** — must move/copy/remove manually |
| **Gitignored, generated** | `.cco/docker-compose.yml`, `.cco/managed/`, `.cco/meta` | Stay on disk | No — regenerated by `cco start` |
| **Gitignored, sensitive** | `secrets.env`, `*.key` | Stay on disk | **Yes** — must handle carefully |
| **Gitignored, ephemeral** | `.tmp/`, `rag-data/` | Stay on disk | No — disposable |

**Key insight**: `git checkout` only affects tracked files. Gitignored files
(claude-state, secrets.env, generated files) **persist across branch switches**.
Profile operations must handle portable gitignored files explicitly.

### 2.3 Portable Gitignored Files — The Hard Problem

When a user does `move project org-a-api --to org-a`:

1. **Tracked files** (`project.yml`, `.claude/`, `memory/`): git handles via
   `git rm` on source + `git checkout target -- path/` on destination
2. **claude-state/** (session transcripts): Contains `/resume` history.
   Must be physically moved/copied between profile-specific storage locations
3. **secrets.env**: Contains API keys. Must be moved with the project —
   user expects secrets to follow the project
4. **Generated files** (docker-compose.yml, managed/, meta): Can be left
   behind — regenerated on next `cco start`

### 2.4 Solution: Shadow Directory for Gitignored Files

Profile-exclusive projects need a **shadow directory** that stores their
gitignored files when they're "not active" (i.e., the user is on a different
profile branch). This is necessary because:

- Git checkout doesn't touch gitignored files
- Without a shadow dir, switching from profile A to B leaves A's gitignored
  files on disk, mixed with B's tracked files
- Returning to A needs those gitignored files back in place

**Proposed structure** — a `.cco/profile-state/` directory at vault root:

```
user-config/
├── .cco/
│   └── profile-state/              # NEW — gitignored
│       ├── org-a/
│       │   └── projects/
│       │       └── org-a-api/
│       │           ├── .cco/claude-state/
│       │           ├── secrets.env
│       │           └── .cco/meta
│       └── personal/
│           └── projects/
│               └── personal-blog/
│                   └── ...
```

**Lifecycle**:
- `profile switch org-a → personal`:
  1. Stash org-a's gitignored files to `.cco/profile-state/org-a/`
  2. `git checkout personal`
  3. Restore personal's gitignored files from `.cco/profile-state/personal/`
- `move project X --to org-a`:
  1. Move tracked files via git (rm source, checkout on target)
  2. Move gitignored files to `.cco/profile-state/org-a/projects/X/`

**Which files to stash**: Only files matching the gitignore patterns that are
"portable" (user data). Generated/ephemeral files can be left or deleted.

Portable gitignored patterns:
- `projects/*/.cco/claude-state/` — session history
- `projects/*/secrets.env` — secrets (and `*.env`, `*.key`, `*.pem`)
- `projects/*/.cco/meta` — schema version (small, useful to preserve)

Non-portable (skip):
- `projects/*/.cco/docker-compose.yml` — regenerated
- `projects/*/.cco/managed/` — regenerated
- `projects/*/.tmp/` — ephemeral
- `projects/*/rag-data/` — large, ephemeral
- `projects/*/.claude/.cco/pack-manifest` — regenerated

## 3. Back-Merge: Shared Resources from Main to Profiles

### 3.1 The Problem

Profile branches are created from main and contain a **superset** of main
(shared + exclusive). When shared resources change on main (e.g., user edits
`global/.claude/CLAUDE.md` while on main), profile branches don't automatically
get those changes.

### 3.2 Current Approach: Sync at Push/Pull Time

The current implementation syncs shared resources during `vault push` and
`vault pull` only:

- **Push**: profile → main (copies shared files that changed on profile)
- **Pull**: main → profile (copies shared files that changed on main)

This works for remote sync but has gaps:

1. **Local-only usage**: Users who don't use a remote never sync. Shared
   changes on main never reach profile branches and vice versa.
2. **Switch timing**: Switching from main to a profile doesn't trigger sync.
   User edits global config on main, switches to profile — old global config.

### 3.3 Proposed Approach: Sync at Switch Time

Add shared resource sync to `profile switch`:

```
cco vault profile switch org-a
  1. Auto-commit pending changes on current branch
  2. git checkout org-a
  3. Stash/restore gitignored files (§2.4)
  4. Sync shared resources from main → org-a (selective checkout)
  5. If conflicts → interactive resolution (L/R/M/D)
  6. Commit sync if changes
```

**Triggers for shared sync**:
- `vault profile switch` — always sync shared from main
- `vault push` — sync shared from profile to main (existing)
- `vault pull` — sync shared from main to profile (existing)
- `vault profile switch main` → then back — manual workaround

**Not triggered by**:
- `vault sync` — only commits, doesn't cross-branch sync
- `cco start` — too implicit, could surprise users

### 3.4 What Counts as "Shared"

Shared resources (always on main, replicated to all profiles):
- `global/` — global Claude config
- `templates/` — project templates
- Packs NOT listed in any profile's `sync.packs`
- `manifest.yml`
- `.gitignore`

**Decision point**: Should ALL profiles see the same shared packs? Currently
yes — a pack is either on main (shared) or on exactly one profile (exclusive).
This is correct per D9 in the original analysis.

### 3.5 Conflict Scenarios During Sync

| Scenario | Example | Resolution |
|----------|---------|------------|
| Only main changed | User edited global CLAUDE.md on main | Auto-copy to profile |
| Only profile changed | User edited global CLAUDE.md on profile | No sync needed (will push to main later) |
| Both changed | Edited on main AND on profile | Interactive prompt (L/R/M/D) |
| New file on main | New shared pack added on main | Auto-copy to profile |
| File deleted on main | Shared pack removed from main | Prompt: delete from profile too? |

### 3.6 Full Merge vs. Selective Checkout (Reconfirmed)

Full `git merge main` into profile is still incorrect because:
- It would try to delete profile-exclusive projects that don't exist on main
- It would create merge commits that complicate the history
- Selective checkout of shared paths only is the correct approach

The existing `_sync_shared_from_default` implementation is correct in approach
but needs adjustment: it currently only runs during `vault pull` and reads from
`origin/main`. It should also work locally (comparing local main branch) and
trigger on `profile switch`.

## 4. CLI UX Redesign

### 4.1 Current Command Structure

```
cco vault profile create <name>
cco vault profile list
cco vault profile show
cco vault profile switch <name>
cco vault profile rename <new-name>
cco vault profile delete <name>
cco vault profile add <project|pack> <name>
cco vault profile remove <project|pack> <name>
cco vault profile move <project|pack> <name> --to <target>
```

**Problems**:
- `cco vault profile move project org-a-api --to org-a` — 8 tokens, very long
- `add` is ambiguous (copy or assign?)
- Inconsistent: `add/remove` act on current profile, `move` requires `--to`
- `profile` is always required even for the most common operations

### 4.2 Option A: Shorter Aliases via `cco vault` Directly

Add shortcuts at `cco vault` level for the most common operations:

```bash
# Profile management (unchanged)
cco vault profile create <name>
cco vault profile list
cco vault profile show
cco vault profile delete <name>
cco vault profile rename <new-name>

# Switch (promoted to vault level — most used operation)
cco vault switch <name>              # alias for vault profile switch

# Resource operations (promoted to vault level)
cco vault move project <name> --to <target>
cco vault copy project <name> --to <target>
cco vault remove project <name>
```

**Pros**: Common ops are shorter. `cco vault move project X --to Y` (7 tokens)
vs. `cco vault profile move project X --to Y` (8 tokens).

**Cons**: Still long. Mixes profile management and resource operations at
vault level.

### 4.3 Option B: Positional Target (No `--to` Flag)

Replace `--to` with positional arguments:

```bash
cco vault move project <name> <target>
cco vault copy project <name> <target>
cco vault remove project <name>
cco vault switch <name>
```

Examples:
```bash
cco vault move project org-a-api org-a      # 6 tokens
cco vault copy pack python-tools personal   # 6 tokens
cco vault remove project old-project        # 5 tokens
cco vault switch org-a                      # 4 tokens
```

**Pros**: Shorter, no flags. Reads naturally: "move project X to Y".
**Cons**: Target position could be ambiguous — is the last argument the project or the
destination? Mitigated by requiring `project` or `pack` keyword.

### 4.4 Option C: Slash-Separated Resource Identifier

Use `<type>/<name>` as a single token:

```bash
cco vault move project/org-a-api org-a
cco vault copy pack/python-tools personal
cco vault remove project/old-project
```

**Pros**: Even shorter (5 tokens for move). Clear that `project/org-a-api`
is the resource.
**Cons**: Non-standard CLI pattern. Could confuse users expecting spaces.

### 4.5 Option D: `cco profile` Top-Level Command

Profile operations are common enough to warrant a top-level command:

```bash
# Profile management
cco profile create <name>
cco profile list
cco profile show
cco profile switch <name>            # or just: cco profile <name>
cco profile delete <name>
cco profile rename <new-name>

# Resource operations
cco profile move project <name> <target>
cco profile copy project <name> <target>
cco profile remove project <name>
```

**Pros**: Shortest paths. `cco profile move project X Y` (6 tokens).
`cco profile switch X` (4 tokens). Separates profile ops from vault ops
(vault = versioning, profile = workspace management).

**Cons**: Profiles are fundamentally a vault feature (git branches). Moving
them out of `vault` namespace may confuse the mental model.

### 4.6 Option E: Hybrid — `cco vault` Shortcuts + Positional

Keep `cco vault profile` for CRUD, promote resource ops + switch to `cco vault`:

```bash
# Profile CRUD (less frequent)
cco vault profile create <name>
cco vault profile list
cco vault profile show
cco vault profile delete <name>
cco vault profile rename <new-name>

# Daily operations (promoted + positional)
cco vault switch <name>                       # 4 tokens
cco vault move project <name> <target>        # 6 tokens
cco vault copy project <name> <target>        # 6 tokens
cco vault remove project <name>               # 5 tokens
```

**Pros**: Best of both worlds. Infrequent CRUD stays namespaced. Daily ops
are short and consistent. Positional target reads naturally.

**Cons**: Two levels of commands (`vault profile create` vs `vault move`)
could be confusing. Mitigated by clear help text and consistent pattern.

### 4.7 Recommendation: Option E (Hybrid)

Option E provides the best balance:
- Profile CRUD: `cco vault profile <cmd>` — infrequent, clear namespace
- Daily ops: `cco vault <move|copy|remove|switch>` — short, consistent
- Positional target: no `--to` flag, reads naturally
- Backward compatibility: `cco vault profile move` can remain as alias

## 5. Resource Operations — Detailed Semantics

### 5.1 `move project <name> <target>`

**Purpose**: Transfer a project from the current branch to `<target>` branch.
The project is removed from source and added to target.

```
Preconditions:
  - Vault initialized, on a branch (profile or main)
  - Project exists on current branch: projects/<name>/
  - Target is a valid profile name or "main"
  - Target != current branch

Steps:
  1. Auto-commit pending changes
  2. Detect if target branch already has projects/<name>/
     → If yes AND files differ: prompt for conflict resolution
     → If yes AND files identical: skip copy, proceed with rm
  3. Show action summary:
       Moving project 'org-a-api':
         From: main (shared)
         To:   org-a (profile)
         Tracked files: 12 files (.claude/, project.yml, memory/)
         Untracked files: claude-state/ (3.2 MB), secrets.env
       Proceed? [y/N]
  4. If confirmed:
     a. Copy tracked files to target:
        git stash (if dirty)
        git checkout <target>
        git checkout <source> -- projects/<name>/
        Update .vault-profile on target (add to sync.projects)
        git add + commit on target
        git checkout <source>
        git stash pop
     b. Remove tracked files from source:
        git rm -r projects/<name>/
        Update .vault-profile on source (remove from sync.projects) if on profile
        git commit
     c. Move portable gitignored files:
        If currently on source: mv files to .cco/profile-state/<target>/
        If not: adjust shadow directory mappings
  5. Output: "Moved project 'org-a-api' to profile 'org-a'"
```

**Rollback**: If any step fails after partial execution:
- Tracked files: `git checkout` can restore from either branch
- Gitignored files: if mv failed mid-way, files are still in source location
- The auto-commit at step 1 ensures no uncommitted work is lost

### 5.2 `copy project <name> <target>`

**Purpose**: Duplicate a project to `<target>` branch. The project exists on
both branches independently (can diverge).

```
Preconditions:
  - Same as move
  - Target branch does NOT already have this project
    (if it does, prompt: "Project already exists on <target>. Overwrite? [y/N]")

Steps:
  1. Auto-commit
  2. Show summary (similar to move, but "Copy" instead of "Move")
  3. If confirmed:
     a. Copy tracked files to target (same as move step 4a)
     b. Do NOT remove from source (unlike move)
     c. Copy portable gitignored files to .cco/profile-state/<target>/
  4. Output: "Copied project 'org-a-api' to profile 'org-a'"

Warning:
  "The project now exists on both branches and may diverge independently."
```

### 5.3 `remove project <name>`

**Purpose**: Delete a project from the current branch. The project ceases to
exist on this profile.

```
Preconditions:
  - Project exists on current branch

Steps:
  1. Auto-commit
  2. Check if project exists on any other branch
     → If not: STRONG WARNING — this is the last copy
  3. Show summary:
       Removing project 'org-a-api' from profile 'org-a':
         Tracked files: 12 files (will be deleted from git)
         Untracked files: claude-state/, secrets.env (will be deleted)
         !! This project does NOT exist on any other branch !!
       Proceed? [y/N]
     OR:
       Removing project 'org-a-api' from profile 'org-a':
         Tracked files: 12 files (will be deleted from git)
         Untracked files: claude-state/, secrets.env (will be deleted)
         This project also exists on: main
       Proceed? [y/N]
  4. Suggest: "Run 'cco vault save' first to save current state"
  5. If confirmed:
     a. git rm -r projects/<name>/
     b. Update .vault-profile (remove from sync.projects)
     c. git commit
     d. Delete portable gitignored files (or move to trash?)
  6. Output: "Removed project 'org-a-api' from profile 'org-a'"
```

**Safety**: `remove` is destructive. The confirmation prompt must clearly
show whether other copies exist.

### 5.4 Pack Operations

Same semantics as project operations. Packs have fewer gitignored files
(only `.cco/install-tmp/` which is ephemeral). The main difference:

- Packs default to shared (on main) — removing from main makes it
  exclusive-only
- Projects are always on exactly one branch

### 5.5 Confirmation and Safety

All three operations (move, copy, remove) should:

1. **Show a summary** of what will happen before executing
2. **Require explicit confirmation** (`[y/N]`, default No)
3. **Suggest `vault sync` first** for operations that delete data
4. **Check for last-copy scenarios** (especially remove)
5. **Support `--yes` flag** to skip confirmation (for scripts)
6. **Require TTY** — refuse to run in non-interactive mode without `--yes`

## 6. Profile Switch — Enhanced Flow

### 6.1 Current Switch Flow

```bash
cmd_vault_profile_switch() {
    _vault_auto_commit
    git checkout "$name" -q
    ok "Switched to profile '$name'"
}
```

### 6.2 Proposed Switch Flow

```
cco vault switch <name>

Steps:
  1. Auto-commit pending changes on current branch
  2. Identify current profile's exclusive projects and packs
  3. Stash gitignored portable files for current profile:
     For each exclusive project on current profile:
       mv projects/<name>/.cco/claude-state/ → .cco/profile-state/<current>/projects/<name>/
       mv projects/<name>/secrets.env → .cco/profile-state/<current>/projects/<name>/
       (same for other portable patterns)
     For each exclusive pack on current profile:
       (similar, fewer files)
  4. git checkout <target>
  5. Restore gitignored portable files for target profile:
     For each exclusive project on target profile:
       mv .cco/profile-state/<target>/projects/<name>/* → projects/<name>/
     For each exclusive pack:
       (similar)
  6. Sync shared resources from main (if target is a profile branch):
     Compare shared paths between target branch and main
     If differences exist:
       Auto-copy files only changed on main
       Interactive prompt for files changed on both sides
       Commit sync
  7. Output:
     "Switched to profile 'org-a'"
     "  Synced 2 shared resources from main"
     "  3 exclusive projects available"
```

### 6.3 Switch to Main

Switching to main is special:
- Main has no exclusive resources (only shared)
- Gitignored files for the previous profile are stashed
- No shared sync needed (main IS the shared source)

```
cco vault switch main

Steps:
  1. Auto-commit
  2. Stash current profile's gitignored files
  3. git checkout main
  4. Output: "Switched to main (shared resources only)"
```

### 6.4 Edge Case: Untracked Project Directories

After `move project X --to org-a` from main, `projects/X/` is `git rm`-ed
from main. But if the directory had gitignored files, those remain on disk
(git doesn't delete gitignored files).

**Solution**: The move operation must explicitly delete/stash these remnant
directories. The shadow directory (`.cco/profile-state/`) solves this.

### 6.5 Edge Case: First Switch After Profile Create

When creating a profile, all resources start on both main and the new profile
(since the branch is created from main). The user needs to `move` resources
to segregate them. Until they do, switching has no visible effect.

This is expected behavior. The workflow is:
1. `cco vault profile create org-a`
2. `cco vault move project org-a-api org-a` (moves to org-a)
3. `cco vault move project org-a-frontend org-a` (moves to org-a)
4. Now switching between main and org-a shows different projects

### 6.6 Performance Considerations

- Stash/restore of gitignored files: `mv` is fast for small files. For large
  `claude-state/` directories (many session transcripts), consider `mv` at
  directory level (O(1) on same filesystem).
- Shared sync: only triggered if shared files actually differ. The `git diff`
  check is fast.
- Branch checkout: git is fast for branch switches.

## 7. Data Integrity and Recovery

### 7.1 Atomicity of Operations

Profile operations involve multiple steps (git ops + file moves). If
interrupted mid-operation:

| Failure point | State | Recovery |
|--------------|-------|----------|
| After git commit on target, before git rm on source | Project exists on both branches | Safe — user can re-run move or manually clean up |
| After git rm on source, before file move | Tracked files gone from source, gitignored files orphaned | Git reflog preserves source state; gitignored files still on disk |
| During gitignored file move | Partial move — some files in source, some in shadow | Files are never deleted, only moved. User can find them in either location |
| During shared sync | Partial sync | Git reflog + stash preserves state |

**Key safety property**: No operation deletes gitignored files permanently.
They are always moved (to shadow dir or to target location). The only
truly destructive operation is `remove`, which requires explicit confirmation.

### 7.2 Backup Before Destructive Operations

For `remove` (and optionally `move`), offer automatic backup:

```bash
# Before remove:
info "Creating backup..."
tar czf ".cco/backups/projects-$name-$(date +%Y%m%d-%H%M%S).tar.gz" \
    "projects/$name/"
```

The `.cco/backups/` directory is gitignored and provides last-resort recovery.
This could be opt-out via `--no-backup` flag.

### 7.3 Operation Log

Maintain a simple operation log at `.cco/profile-ops.log` (gitignored):

```
2026-03-24T10:15:00 MOVE project org-a-api main→org-a (commit abc1234)
2026-03-24T10:16:00 MOVE project org-a-frontend main→org-a (commit def5678)
2026-03-24T10:20:00 SWITCH main→org-a
```

Useful for debugging and understanding what happened.

## 8. Shared Resource Sync — Unified Model

### 8.1 Sync Triggers (Updated)

| Trigger | Direction | Scope |
|---------|-----------|-------|
| `vault push` (on profile) | Profile → main | Shared files changed on profile |
| `vault pull` (on profile) | Main → profile | Shared files changed on main |
| `vault switch` (to profile) | Main → profile | Shared files changed on main |
| `vault switch` (to main) | None | Main is the source of truth |
| `vault sync` | None | Local commit only, no cross-branch |

### 8.2 Sync Algorithm (Unified)

```bash
_sync_shared_to_profile() {
    local vault_dir="$1" profile_branch="$2"
    local default_branch=$(_vault_default_branch)

    # Find shared paths
    local shared_paths=($(_list_shared_paths "$vault_dir"))

    # Compare profile vs local main (not origin/main)
    local changed_files
    changed_files=$(git -C "$vault_dir" diff "$default_branch" "$profile_branch" \
        --name-only -- "${shared_paths[@]}")

    [[ -z "$changed_files" ]] && return 0

    # For each changed file, determine who changed it
    local merge_base
    merge_base=$(git -C "$vault_dir" merge-base "$default_branch" "$profile_branch")

    while IFS= read -r file; do
        local changed_on_main changed_on_profile
        changed_on_main=$(git -C "$vault_dir" diff "$merge_base" "$default_branch" \
            --name-only -- "$file")
        changed_on_profile=$(git -C "$vault_dir" diff "$merge_base" "$profile_branch" \
            --name-only -- "$file")

        if [[ -n "$changed_on_main" && -n "$changed_on_profile" ]]; then
            # Both changed → conflict resolution
            _resolve_shared_conflict ...
        elif [[ -n "$changed_on_main" ]]; then
            # Only main changed → auto-copy to profile
            git checkout "$default_branch" -- "$file"
        fi
        # If only profile changed → no action (profile's version is newer)
    done <<< "$changed_files"
}
```

### 8.3 When Main Changes Affect Running Sessions

If a user is in a `cco start` session and someone (or another session)
modifies shared resources on main, the running session doesn't see those
changes (Docker mounts are live but the vault branch hasn't synced).

This is acceptable — changes are picked up at next `vault switch` or
`vault pull`. No real-time sync is needed.

---

## 9. Migration Path

### 9.1 Resetting Existing Profiles

Users who created profiles under the tracking-only model need to reset:

```bash
# Reset to pre-profile state (on user's host machine)
cd ~/.config/claude-orchestrator  # or wherever user-config lives
git checkout main
git branch -D org-a personal org-b    # delete profile branches
```

All projects remain on main. User can then use the new commands to
properly move resources to profiles.

### 9.2 Backward Compatibility

- Vaults without profiles: completely unchanged (no migration needed)
- Vaults with profiles (tracking-only): must reset and re-create
  (no automatic migration — the tracking-only model has no reliable
  state to migrate from since all files exist on all branches)

### 9.3 New Vault .gitignore Additions

```gitignore
# Profile state — gitignored files stashed during profile switch
.cco/profile-state/

# Profile operation backups
.cco/backups/

# Profile operation log
.cco/profile-ops.log
```

## 10. Open Questions for Design Phase

### Q1: Shadow Directory vs. Cleanup-on-Switch

**Option A** (Shadow directory — recommended): Stash gitignored files to
`.cco/profile-state/` during switch. More complex but preserves all data.

**Option B** (Cleanup on switch): Delete gitignored files on switch,
regenerate on next `cco start`. Simpler but loses `claude-state/` (session
history) and `secrets.env`.

**Recommendation**: Option A. Users expect session history and secrets to
persist across profile switches.

### Q2: Should `remove` Create a Backup?

**Option A**: Always backup to `.cco/backups/` before remove.
**Option B**: Backup only if project doesn't exist on another branch.
**Option C**: Never backup — user should use `vault sync` before remove.

**Recommendation**: Option B — auto-backup when it's the last copy. For
other cases, the confirmation prompt is sufficient since the user can
recover from git or from the other branch.

### Q3: Should `copy` Be Exposed at All?

`copy` creates divergence risk (same project on two branches). Is this
actually useful? Use cases:
- Testing a project modification without affecting the original
- Sharing a project template across profiles

**Alternative**: Users can achieve this with
`cco vault move project X Y && cco vault copy project X main`

**Recommendation**: Keep `copy` but mark it as advanced. The primary
workflow uses `move` and `remove` only.

### Q4: CLI Namespace — Option E Confirmed?

Need user confirmation on the CLI structure before proceeding to design.
See §4.7 for the recommended approach.

### Q5: Projects on Main — Default Behavior

With real isolation, what happens to projects on main that are NOT assigned
to any profile? Options:

**Option A**: Projects on main are visible on ALL profiles (shared projects).
This is consistent with how packs work but complicates the model — profile
branches must keep main's projects in sync.

**Option B**: Projects on main are visible ONLY when on main. Profiles only
see their exclusive projects. This is simpler and cleaner.

**Option C**: Hybrid — projects can be explicitly marked as "shared" in main
(like packs), but default to main-only.

**Recommendation**: Option B. Projects are almost always profile-exclusive.
If a user wants a project on multiple profiles, they use `copy`. Main is for
shared config (global, templates, packs), not for projects.

If Option B is chosen, `vault profile create` should warn when there are
projects still on main: "You have 5 projects on main. Use 'cco vault move'
to assign them to profiles."

### Q6: Pack Sharing Model

Currently: packs NOT listed in any profile's `sync.packs` are "shared"
(visible everywhere). This is the correct model and should remain.

With real isolation, shared packs (on main) must be synced to profile
branches — same as global config. The existing sync mechanism handles this.

Profile-exclusive packs work like projects: exist only on one branch.

## Addendum: Decisions Revised After Initial Analysis

The following decisions were revised during the design discussion that followed
this analysis. The definitive design is in `profile-isolation-design.md`, which
takes precedence.

| Topic | Original (this analysis) | Revised (design doc) |
|-------|--------------------------|----------------------|
| Commit command name | `vault sync` — commit config changes | Renamed to `vault save` (§ in design doc) |
| `vault save` propagation | Proposed sync at switch time only | `vault save` propagates shared resources to main AND to all profiles immediately, not just at switch time |
| Switch precondition | Auto-commit pending changes before switch | Switch requires a clean working tree — no auto-commit. User must run `vault save` manually before switching |
| `copy` command | Proposed as advanced feature, kept for v1 (§10 Q3) | Deferred — not included in initial release |
| Merge-base advancement | Not addressed in this analysis | After shared sync, the merge-base is advanced via `git merge -s ours` to prevent repeated conflict prompts for already-resolved divergences |
