# Managed Integrations — Protocol & Guide

> Audience: claude-orchestrator maintainers adding new built-in integrations.
> Related: [architecture.md](./architecture.md) (ADR section 6)

---

## What is a "Managed Integration"?

A **managed integration** is a framework capability that:

1. Requires runtime configuration generated from `project.yml` values
2. Benefits from framework-provided extras (port conflict resolution, lifecycle management)
3. Should not pollute the user's `mcp.json` or `.claude/` space

Examples of managed integrations (current and planned):

| Integration | Status | Description |
|-------------|--------|-------------|
| Browser MCP | Implemented | chrome-devtools-mcp with CDP proxy and port resolution |
| GitHub MCP | Implemented | Managed GitHub server with token env configuration |
| RAG | Planned | Vector DB sidecar + MCP server |

**What is NOT a managed integration**: anything the user configures themselves via
`mcp.json` or `project.yml` without framework extras. GitHub MCP configured by the user
in `mcp.json` is user-managed, not framework-managed.

## Host Layout

```
projects/
  my-project/
    project.yml           ← USER: source of truth
    docker-compose.yml    ← AUTO-GEN: generated by cco start
    mcp.json              ← USER: custom MCP servers (optional)
    secrets.env           ← USER: project secrets
    setup.sh              ← USER: runtime setup script
    .claude/              ← USER: CLAUDE.md, rules, agents, skills, settings
    claude-state/         ← FRAMEWORK: session transcripts + auto memory
    .managed/             ← FRAMEWORK: managed integration configs (never edit)
      browser.json        ←   Browser MCP config (generated by cco start)
      .browser-port       ←   Effective CDP port (runtime state)
      # future: github.json, rag.json, ...
```

`.managed/` is gitignored (migration 003 adds this automatically). All contents are
regenerated by `cco start` and cleaned up by `cco stop`.

## Container Layout

```
/workspace/
  .managed/               ← MOUNT: projects/<name>/.managed/:ro
    browser.json          ←   read by entrypoint for merge
  .mcp.json               ← MOUNT: projects/<name>/mcp.json:ro (user)
  project.yml             ← MOUNT: read-only
  .claude/                ← MOUNT: projects/<name>/.claude/ (user)
  <repos>/                ← MOUNT: repos

~/.claude.json            ← RUNTIME: rebuilt each session from:
                          ←   global mcp → project mcp → .managed/*.json
~/.claude/                ← MOUNT: global/.claude/ (user)
```

## 9-Step Protocol for New Integrations

Follow these steps in order. Skip optional steps if not needed.

### Step 1 — Declare in project.yml (Required)

Add a top-level section to the project.yml template and schema:

```yaml
# defaults/_template/project.yml
my-integration:
  enabled: false
  # integration-specific options...
```

Parse in `cmd-start.sh` in the "integration config" section:

```bash
local myint_enabled
myint_enabled=$(yml_get "$project_yml" "my-integration.enabled")
[[ "$myint_enabled" != "true" ]] && myint_enabled="false"
```

### Step 2 — Generate managed config (Required if has MCP)

Add a `_generate_<integration>()` function in `cmd-start.sh`:

```bash
_generate_myintegration() {
    local out_file="$1"
    # ... build the JSON ...
    printf '{"mcpServers": {"my-server": {...}}}\n' > "$out_file"
}
```

Call it in the generation block:

```bash
if [[ "$myint_enabled" == "true" ]]; then
    mkdir -p "$project_dir/.managed"
    _generate_myintegration "$project_dir/.managed/myintegration.json" ...
else
    rm -f "$project_dir/.managed/myintegration.json"
fi
```

### Step 3 — Mount in docker-compose (Required)

The `.managed/` directory is already mounted generically — **no compose change needed**:

```bash
# Already in cmd-start.sh compose generation:
if [[ -d "$project_dir/.managed" ]] && [[ -n "$(ls -A "$project_dir/.managed" 2>/dev/null)" ]]; then
    echo "      - ./.managed:/workspace/.managed:ro"
fi
```

### Step 4 — Merge in entrypoint (Required if has MCP)

The entrypoint already loops over all `*.json` files in `/workspace/.managed/` —
**no entrypoint change needed**:

```bash
# Already in config/entrypoint.sh:
if [ -d "/workspace/.managed" ]; then
    for mcp_file in /workspace/.managed/*.json; do
        [ -f "$mcp_file" ] || continue
        # conflict warning + merge...
    done
fi
```

### Step 5 — Extra Docker config (Optional)

If the integration needs Docker-level configuration, add conditionally in the compose
generation block:

- `extra_hosts` — needed when the container must reach the host (e.g., socat proxy)
- `environment` — pass runtime values (`CDP_PORT`)
- Volume mounts — e.g., Docker socket
- Port mappings

### Step 6 — Runtime process (Optional)

If the integration requires a sidecar process inside the container, start it in
`config/entrypoint.sh` conditioned on the managed config file:

```bash
if [ -f "/workspace/.managed/myintegration.json" ]; then
    my-sidecar-process &
    echo "[entrypoint] my-integration sidecar started" >&2
fi
```

### Step 7 — Port conflict resolution (Optional)

If the integration occupies a host port:

```bash
_resolve_myintegration_port() {
    local preferred="$1"
    local current_project="$2"
    # scan running sessions, auto-assign if conflict
    # save effective port to .managed/.<integration>-port
}
```

### Step 8 — Cleanup (Recommended)

In `cmd-stop.sh`, clean managed files when stopping a named project:

```bash
rm -f "$project_dir/.managed/myintegration.json"
rm -f "$project_dir/.managed/.<integration>-port"
```

In `cmd-start.sh`, clean when `enabled=false` (the `else` branch of Step 2).

### Step 9 — Session-level CLI flags (Optional)

If the integration can be toggled at runtime without editing `project.yml`,
add `--<integration>` / `--no-<integration>` flags to `cco start`.

**Pattern in `cmd-start.sh`:**

```bash
# 1. Declare session override variable (empty = unset)
local opt_myint=""   # "on" | "off" | "" (unset = read from project.yml)

# 2. Parse in the flag-handling loop
--my-integration)    opt_myint="on";  shift ;;
--no-my-integration) opt_myint="off"; shift ;;

# 3. Document in the help text
--my-integration     Enable <integration> for this session only
--no-my-integration  Disable <integration> for this session only

# 4. Apply after reading project.yml values
[[ "$opt_myint" == "on"  ]] && myint_enabled="true"
[[ "$opt_myint" == "off" ]] && myint_enabled="false"
```

The note shown in `--help` is standard and should be reproduced verbatim:

```
Session flags (--chrome, --no-chrome, --no-docker) override project.yml for
one session only. To change the default, edit project.yml instead.
```

Add the new flags to that line.

**When to use this step:**
- Integration can reasonably be toggled per-session (e.g., browser, docker socket)
- Skip if the integration is always-on, or if enabling it requires build-time changes

## Conflict Policy

When a managed MCP server key conflicts with a user-configured key (same name in both
`mcp.json` and a `.managed/*.json`), the **managed version wins** (last-write in the
merge loop). The entrypoint emits a visible warning:

```
[entrypoint] WARNING: managed MCP 'chrome-devtools' overrides user-configured server
```

The user should remove the duplicate from their `mcp.json`. The framework does not
block startup — just warns.

## Reference Implementation: Browser MCP

The Browser integration is the canonical example of all 9 steps:

| Step | File | What it does |
|------|------|-------------|
| 1 | `defaults/_template/project.yml` | `browser: {enabled, cdp_port, mcp_args}` |
| 2 | `lib/cmd-start.sh` `_generate_browser_mcp()` | Writes `.managed/browser.json` |
| 3 | `lib/cmd-start.sh` (compose gen) | `.managed/` dir mount (generic) |
| 4 | `config/entrypoint.sh` | Generic loop merges `browser.json` |
| 5 | `lib/cmd-start.sh` (compose gen) | `extra_hosts`, `CDP_PORT` env |
| 6 | `config/entrypoint.sh` | socat CDP proxy conditioned on `.managed/browser.json` |
| 7 | `lib/cmd-start.sh` `_resolve_browser_port()` | Port conflict scan; writes `.managed/.browser-port` |
| 8 | `lib/cmd-stop.sh` | Removes `browser.json` and `.browser-port` from `.managed/` |
| 9 | `lib/cmd-start.sh` flags | `--chrome` / `--no-chrome` session overrides |

## Checklist for New Integrations

- [ ] Added `integration-name:` section to `defaults/_template/project.yml`
- [ ] Created `migrations/project/NNN_*.sh` if project.yml format changed
- [ ] Added `_generate_<integration>()` function in `cmd-start.sh`
- [ ] Generation call with `mkdir -p .managed` guard
- [ ] Cleanup call in the `else` branch (disabled) and `cmd-stop.sh`
- [ ] Extra Docker config added conditionally to compose generation (if needed)
- [ ] Runtime sidecar started in entrypoint conditioned on managed config file (if needed)
- [ ] Port conflict resolution implemented (if needed)
- [ ] Session flags `--<integration>` / `--no-<integration>` added to `cco start` (if needed)
- [ ] Tests added for: enabled, disabled, cleanup, compose mount, session flag overrides
- [ ] `.managed/` in project `.gitignore` (migration 003 handles existing projects)
