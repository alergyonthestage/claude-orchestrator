## 1. Create a Project

```bash
cco project create my-app \
  --repo ~/projects/backend-api \
  --repo ~/projects/frontend-app \
  --description "My SaaS application"
```

This generates the structure in `projects/my-app/`:

```
projects/my-app/
├── project.yml              # Main configuration
├── .claude/
│   ├── CLAUDE.md            # Instructions for Claude (project-level)
│   ├── settings.json        # Settings overrides (optional)
│   ├── agents/              # Custom subagents (optional)
│   └── rules/               # Custom rules (optional)
├── claude-state/            # Claude state: auto memory + session transcripts (persists across sessions and rebuilds)
│   └── memory/              # Auto memory subdir
└── docker-compose.yml       # Auto-generated by `cco start`
```

If you pass `--repo`, the CLI auto-detects basic information from the repositories (package.json, pyproject.toml, go.mod, etc.) and populates CLAUDE.md with relevant details.

## 2. Configure project.yml

### Repos vs Extra Mounts vs Packs

| Field | Purpose | Mounted at | Loaded in context | Typical use |
|-------|---------|------------|-------------------|-------------|
| `repos` | Active working repositories | `/workspace/<name>/` | On-demand (when Claude reads files) | Code Claude modifies |
| `extra_mounts` | Reference material | Custom path | No (Claude reads on request) | Shared libs, API specs, datasets |
| `packs` | Reusable cross-project knowledge | `/workspace/.packs/<name>/` | Yes (via `@.claude/packs.md`) | Conventions, business overviews, guidelines |

**repos** — The repositories Claude actively works on. Mounted as subdirectories of `/workspace/` with read-write access. Any `.claude/CLAUDE.md` inside a repo is loaded automatically when Claude reads files in that directory.

**extra_mounts** — Additional material for reference. Mounted at an arbitrary path in the container, typically read-only. Useful for shared libraries (e.g. internal frameworks), API specs, or any reference files Claude might need to read but not modify.

**packs** — Groups of knowledge documents (conventions, business overviews, guidelines) defined once in `global/packs/` and reusable across projects without copying anything. The pack's source directory is mounted read-only and files are automatically injected into Claude's context via `@import`. Ideal for cross-cutting documentation that applies to all projects for a given client or domain.

```yaml
repos:
  - path: ~/projects/backend-api
    name: backend-api
  - path: ~/projects/frontend-app
    name: frontend-app

extra_mounts:
  - source: ~/projects/shared-framework
    target: /workspace/shared-framework
    readonly: true

packs:
  - my-client-knowledge   # → global/packs/my-client-knowledge/pack.yml
```

### Configure a Pack

Create `global/packs/<name>/pack.yml`:

```yaml
name: my-client-knowledge
source: ~/documents/my-client    # directory with knowledge files
target: /workspace/.packs/my-client

files:
  - backend-coding-conventions.md
  - business-overview.md
  - testing-guidelines.md
```

Then add **once** to the project's `projects/<n>/.claude/CLAUDE.md`:

```markdown
@.claude/packs.md
```

Every `cco start` automatically regenerates `.claude/packs.md` with `@import` directives for the pack's files. The original files stay in your knowledge repo — zero duplication.

### Ports and Environment Variables

```yaml
docker:
  ports:
    - "3000:3000"       # Frontend dev server
    - "4000:4000"       # Backend API
    - "5432:5432"       # PostgreSQL (sibling container)
  env:
    NODE_ENV: development
    DATABASE_URL: "postgresql://postgres:postgres@postgres:5432/myapp"
```

Ports make services accessible from `localhost` on macOS. Environment variables are available inside the container.

### Authentication

```yaml
auth:
  method: oauth         # Default: use token from macOS Keychain
  # method: api_key     # Alternative: use ANTHROPIC_API_KEY env var
```

## 3. Writing a Good CLAUDE.md

The file `projects/<name>/.claude/CLAUDE.md` is the central place to give Claude context about the project. A good CLAUDE.md is the difference between productive sessions and sessions where Claude constantly asks for clarification.

### Recommended Approach: Use /init

On the first session for a project, ask Claude to analyze the codebase:

```
> Use /init to analyze the repositories and generate a detailed CLAUDE.md
```

Claude Code's `/init` command analyzes files, detects structure, frameworks, build/test commands, and automatically generates relevant content. You can then refine it manually.

### What to Include

- **Overview**: What the project does, how the repositories relate to each other
- **Architecture**: Technologies, patterns, main components
- **Commands**: Build, test, dev server, deploy — for each repository
- **Conventions**: Code style, naming, patterns to follow
- **Infrastructure**: If you use Docker Compose for services (postgres, redis...), specify the project network

### Example

```markdown
# Project: my-saas

## Overview
SaaS platform with a Node.js backend and React frontend.

## Repositories
- `/workspace/backend-api/` — REST + GraphQL API (Node.js, Express, Prisma)
- `/workspace/frontend-app/` — React SPA with Vite + TailwindCSS

## Architecture
The backend exposes REST API on :4000 and GraphQL on :4000/graphql.
The frontend communicates via fetch with the backend.
PostgreSQL reachable as `postgres:5432` on the Docker network.

## Commands
### backend-api
- Dev: `npm run dev` (port 4000)
- Test: `npm test` / `npm run test:watch`
- Lint: `npm run lint`

### frontend-app
- Dev: `npm run dev` (port 3000)
- Build: `npm run build`
- Test: `npm run test`

## Infrastructure
Docker network: `cc-my-saas`
For infrastructure docker-compose files, use:
\`\`\`yaml
networks:
  default:
    external: true
    name: cc-my-saas
\`\`\`
```

## 4. Context Hierarchy

Claude Code loads instructions in order of precedence:

```
1. ~/.claude/CLAUDE.md                          ← Global (always loaded)
   └── ~/.claude/rules/*.md

2. /workspace/.claude/CLAUDE.md                 ← Project (always loaded)
   ├── /workspace/.claude/rules/*.md
   └── @.claude/packs.md                        ← Knowledge packs (if configured)
         └── @/workspace/.packs/<name>/*.md

3. /workspace/<repo>/.claude/CLAUDE.md          ← Repository (on-demand)
```

Project settings (level 2) override global settings (level 1). Repository instructions (level 3) are added when Claude reads files in that directory.

**Knowledge packs** are nested at the project level via `@.claude/packs.md` (auto-generated by `cco start`): Claude receives the knowledge files in context without you having to copy them into the project.

For more details on the hierarchy see [context.md](../reference/context.md).

## 5. Using External Directories for Projects

If you want to keep projects in a different directory (e.g. on another disk or to separate them across machines):

```bash
export CCO_PROJECTS_DIR=~/my-custom-projects
cco project create my-app --repo ~/projects/backend
cco start my-app
```

Similarly for global configuration:

```bash
export CCO_GLOBAL_DIR=~/my-claude-config
```

This lets you clone the `claude-orchestrator` repo on multiple machines while keeping projects and configurations separate.

## 6. Versioning User Configuration

Since `global/` and `projects/` are gitignored in the orchestrator repo, you can version them separately.

### "Config repo" Pattern

If you work across multiple machines, keep your user configuration in a separate repo:

```bash
# On the main machine, initialize a repo for the config
cd ~/claude-orchestrator
git init global/
cd global/
git add -A && git commit -m "Initial global config"
git remote add origin <your-config-repo-url>
git push -u origin main
```

On another machine:

```bash
# Clone the orchestrator
git clone <orchestrator-url> ~/claude-orchestrator
cd ~/claude-orchestrator
cco init  # creates global/ from defaults

# Or: replace global/ with your config repo
rm -rf global/
git clone <your-config-repo-url> global/
```

### Managing Defaults Updates

When the tool is updated (`git pull`), the defaults in `defaults/` may change. To see differences compared to your current configuration:

```bash
# Compare your config with the new defaults
diff -r defaults/global/.claude/ global/.claude/

# To reset to new defaults (overwrites customizations):
cco init --force
```

### Practical Example

```bash
# Initial setup
git clone <orchestrator-url> ~/claude-orchestrator
cd ~/claude-orchestrator
cco init                                    # Copies defaults → global/, builds image

# Customize
vim global/.claude/CLAUDE.md                # Your global instructions
vim global/.claude/rules/my-rules.md        # Your custom rules

# Create projects
cco project create my-app --repo ~/code/my-app
vim projects/my-app/.claude/CLAUDE.md       # Instructions for this project

# Update the tool (no conflicts!)
git pull                                    # Updates only defaults/, bin/, docs/
```

## 7. Post-Creation Checklist

After `cco project create`:

- [ ] Verify `project.yml`: repos, ports, environment variables
- [ ] Customize `.claude/CLAUDE.md` (or use `/init` on the first session)
- [ ] Add custom settings in `.claude/settings.json` if needed
- [ ] Configure `packs:` in `project.yml` if you have cross-project knowledge to share
- [ ] If using packs: add `@.claude/packs.md` to the project's `CLAUDE.md` (once only)
- [ ] First run: `cco start <name>` — verify everything works
- [ ] Optional: add custom subagents in `.claude/agents/`
